# 排课规则定义（定稿）

本文件定义排课模块的全部业务规则，是编码与数据层的依据。四项前置问题已经老师确认（见 §0），
本文件据此定稿。状态与流水枚举取值见 `dev-guide.md`，数据表见 `schema.sql`。

---

## 0. 前置决策（老师已确认）

| # | 问题 | 结论 | 对模型/规则的影响 |
|---|---|---|---|
| 1 | 是否有小班课 | **有**。既有 1 对 1，也有小班课（一节课多名学员） | `class_sessions` 与学员**多对多**，加 `session_students` 关联表；循环课同理加 `recurrence_students`；见 §4/§6 |
| 2 | 课时按次还是按时长 | **按次**。一次买 10 节这种，不同时长都扣 1 次 | `credit_logs.delta` 保持 **int**，无需小数 |
| 3 | 课程类型 | **两种，需作为字段存**：补课类 90 分钟 / 全英剑桥课程 120 分钟 | `class_sessions` 加 `course_type` 字段；建课按类型带出默认时长、允许手改；`constants/` 定义课程类型枚举 |
| 4 | 请假规则 | **请假一律不扣课时**，无论提前多久 | §6 的 absent 不再有“扣/不扣”分支；保留 `manual` 手动扣除入口以备特殊情况 |

## 1. 循环课的生成

**规则：循环规则必须带结束日期，创建时一次性生成全部课程实例；生成跨度上限 26 周（约半年）。**

- `recurrences` 存规则（course_type、weekday、time_of_day、duration_min、start_date、end_date）。
  小班课的循环，其固定学员集合存 `recurrence_students`。
- 保存规则时，按 start_date→end_date 之间每个匹配 weekday 生成一批 `class_sessions`
  实例（status=scheduled），每条带 `recurrence_id` 指回规则；同时把该规则的学员集合
  复制进每节课的 `session_students`。
- “长期固定”场景：界面给默认结束日期（如 +16 周），二期再做“延长”。

**取舍理由：** 滚动生成需引入定时任务，一期不值当；一次性生成把每节课落成真实行，
冲突检测与周视图渲染都变成“查已有行”，最简单可靠。强制 end_date + 26 周上限，防止无限生成与表爆量。

---

## 2. 修改单节课，是否影响整个系列

**规则：默认“只改这一节”。系列级修改是另一个入口（编辑循环规则），且只影响未来未上课的实例。**

- **改单节**（时间/时长/备注/学员增减）：只更新那一条 `class_sessions` 及其 `session_students`，
  与系列脱钩，视为该系列的一个“例外”。循环规则与其它实例不受影响。
- **改系列**：走“编辑循环规则”，仅对**未来且 status=scheduled** 的实例生效——删掉这些未来未上课
  实例后按新规则重新生成；已 completed / absent 的历史课程一律不动。

**取舍理由：** 不照搬日历软件“仅此课 / 此课及以后 / 整个系列”三选一（成本高、边界多），
拆成“单节编辑=只影响单节、系列编辑=只影响未来”两个清晰入口，覆盖绝大多数需求且不动历史账目。

---

## 3. 删除循环规则时，已生成的课程如何处理

**规则：只删除未来 status=scheduled 的实例；completed / absent 的历史课程全部保留。**

- 删除规则 → 硬删除其名下**未来且未上课**的 `class_sessions`（连带其 `session_students`）。
  这些课从未扣课时（§6：scheduled 不扣），删除不产生流水。
- 已 completed / absent 的历史实例保留，它们关联 `credit_logs`，删了会破坏账目可追溯。
- 规则行本身软删（加 deleted 标记），保留为历史依据。

**取舍理由：** 课时账本是底线，任何操作不得让历史流水悬空；未来未上课实例无账目牵连，删除零成本。

## 4. 冲突检测标准（小班课已确认）

**规则：老师是唯一资源。同一节课内可有多名学员（小班课），但两节不同的课时间重叠即冲突，默认阻止保存、二次确认可强存。**

- 冲突的是**两节不同的 `class_sessions`**，不是学员：
  - **同一节小班课有多个学员** → 正常，不是冲突（他们在同一 session 内）。
  - **两节不同的课时间重叠** → 冲突（老师分身不了），无论学员是否相同。
- 重叠判定：两节课时间区间 `[开始, 开始+时长)` 相交即冲突，即 `A.start < B.end 且 B.start < A.end`。
  **首尾相接不算冲突**（10:00 结束、10:00 开始，OK）。
- 参与检测的课：status ∈ {scheduled, completed, absent}；cancelled 不占时段。
- **强存**：默认硬阻止并提示与哪节课冲突；允许一次明确的二次确认后强制保存（老师排补课偶尔需要）。

**取舍理由：** 把“冲突”定义在 session 粒度而非学员粒度，天然容纳小班课；老师单资源 + 区间相交是唯一硬规则。
保留二次确认强存，避免老师被逼删课绕路。

## 5. 状态流转与可逆性

**规则：`scheduled` 是中心态，三个出向操作全部可逆；撤销通过写“反向流水”实现，不删原流水。**

```
                标记完成
   scheduled ──────────────► completed
       ▲                         │ 撤销完成
       │◄────────────────────────┘
       │        标记缺勤(请假)
       ├──────────────────► absent
       │◄──────────────────┘ 撤销缺勤
       │        取消
       ├──────────────────► cancelled
       │◄──────────────────┘ 恢复
```

- `scheduled → completed / absent / cancelled` 三个正向操作，每个都**可逆回 scheduled**。
- **不做** completed↔absent 横向直跳，一律“先撤销回 scheduled，再重标”，状态机简单不易错。
- **可逆实现**：凡撤销涉及课时，往 `credit_logs` **追加一条反向记录**（如撤销一个已完成的小班课
  → 给其每个已扣学员各 +1），绝不删改原始流水。`credit_logs` 保持 append-only，余额（delta 累加）自动回正。

**取舍理由：** 单中心态 + 全可逆，误操作可恢复又避免 N×N 跳转组合爆炸；反向记账落地 dev-guide 记账原则。

## 6. 各状态对课时的影响（按次计、请假不扣、小班课逐学员）

**扣减单位固定为“1 次”，与时长/课程类型无关。小班课在完成/撤销时对每个参与学员各记一条流水。**

| 状态 | 课时影响 | 写入 credit_logs |
|---|---|---|
| scheduled | **不扣** | 无 |
| completed | 参与的**每个学员各扣 1 次** | 每学员一条：delta=−1，reason=attend，关联 session_id |
| absent（请假） | **一律不扣** | 无 |
| cancelled | **不扣**（从未上课） | 无 |
| 撤销完成 | 已扣学员**各退 1 次** | 每学员一条：delta=+1，reason=manual，关联同一 session_id |
| 手动调整（特殊情况） | 老师手动增减 | delta=±N，reason=manual，备注原因 |

- **请假一律不扣**：absent 不再有“扣/不扣”选择分支，直接不产生流水。
- **取消不涉及退回**：scheduled 阶段没扣过，取消自然不产生也不退回课时。
- **保留 `manual` 手动入口**：应对将来规则变化或个别特殊情况（如误标已完成后需单独修正某学员），
  不写死成唯一路径。
- **课时只在“完成/撤销/手动”时变动，排课、取消、请假都不动课时**——账本干净的关键。

**取舍理由：** 按次计最贴合“买 10 节”的售卖方式，`delta` 保持 int；请假免扣是老师明确的经营策略，
简化了缺勤逻辑；`manual` 作为兜底入口，保证规则演进时不必改数据结构。

## 7. 课程类型（course_type）

| 类型 | 枚举值（建议） | 默认时长 |
|---|---|---|
| 补课类 | `makeup` | 90 分钟 |
| 全英剑桥课程 | `cambridge` | 120 分钟 |

- 建课时选类型 → 自动带出默认时长（90 / 120），**允许老师手动改**具体时长。
- 类型与默认时长的映射集中定义在 `src/constants/`（如 `CourseType` 枚举 + `COURSE_TYPE_DEFAULT_DURATION`），
  业务代码不写字面量。
- `class_sessions.course_type` 与 `recurrences.course_type` 都存该字段。

## 数据模型影响小结（进入 schema.sql 的依据）

因“小班课”确认为多对多，相比最初设想的表结构有如下调整：

- `class_sessions` **去掉** `student_id`，改由 **`session_students(session_id, student_id)`** 关联；
  **新增** `course_type`。
- `recurrences` 同理去掉单一 `student_id`，其固定学员集合由 **`recurrence_students(recurrence_id, student_id)`** 表达；
  **新增** `course_type`。
- `credit_logs` 保持 `student_id` + `session_id`（一节小班课 → 多条流水，每学员一条），`delta` 保持 int。
- `packages`、`students` 不变。
