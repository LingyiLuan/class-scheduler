# 二期设计文档（待逐条确认，未写代码）

## 定位

- **单工作室工具**，只服务这一家；不做多工作室 SaaS。
- 朋友是这家的 **owner / admin**。
- **轻度预留平台化**：核心表加 `workspaceId`，当前全填同一固定值，查询暂不强制过滤。目的：将来若转平台，隔离逻辑与历史数据迁移成本最低。

本文档逐条给方案与取舍，标 **【待定】** 的地方等你拍板后再开发。

---

## 〇、workspaceId 预留（先定这个，它影响后面所有表）

### 哪些表加
一期已有、且属于"业务数据"的表都加：

`students` `classSessions` `recurrences` `packages` `creditLogs` `courseTypes` `notifications`

二期新增的表（见第八节：`guardianLinks` `inviteCodes` `leaveRequests`）**建表时就带** `workspaceId`，无需回填。

**不加的**：
- `users`：用户是**跨工作室**的概念（平台化后一个人可能属于多个工作室），不适合单个 `workspaceId`。用户与工作室的关系将来用"成员关系"表达（membership），不是一个字段。二期先不动 users 的归属，留到真平台化再设计。
- `notifyLogs`：纯技术日志，可加可不加；建议加（成本为零、口径统一），但不强制。

### 默认值怎么定
**【待定 A】** 两个选项：
- **A1（推荐）不透明常量**：定一个稳定的字符串，如 `WORKSPACE_DEFAULT = 'ws_main'`，前端 `src/constants` 与云端 `_shared` 各存一份（同一值）。工作室身份不绑定到某个人，owner 换人也不受影响。平台化时，这个默认工作室成为"多工作室里的第一个"。
- **A2 用 owner 的 openid 当 workspaceId**：天然唯一。缺点：工作室身份和某个人的微信号绑死，owner 变更/离场就尴尬；语义也怪（工作室 ≠ 人）。

→ 推荐 **A1**：一个稳定、与人解耦的常量。

### 一期存量数据怎么回填
- 一次性迁移（做法同一期"课程类型迁移"）：给上述 7 张表里**缺 `workspaceId`** 的记录，统一写成 `WORKSPACE_DEFAULT`。幂等（只补缺字段）。
- 新写入：所有 `add` 一律带上 `workspaceId`（云端在写入处统一注入）。
- **查询暂不加过滤**：一期就一个工作室，过滤没意义还增加出错面。只"埋字段"，不"用字段"。平台化那天才加 `where(workspaceId)` + 成员校验。

### 取舍
现在埋：每条写入多一个字段，几乎零成本。现在不埋：将来平台化要猜哪些历史行属于哪个工作室、或全表停机迁移。**埋 workspaceId 是这次预留里唯一"必须现在做"的**——其它平台化改造都能等。

## 一、权限模型

### 1. 角色能做什么（owner=admin / teacher / student）

| 能力 | owner（admin） | teacher | student（家长） |
|---|---|---|---|
| 看数据范围 | **全工作室所有人**的数据 | **仅自己名下**（ownerId=自己） | **仅关联孩子**的 |
| 建/改/删 学员 | ✅ | ✅（自己的） | ❌ |
| 排课 / 改期 / 取消 | ✅ | ✅（自己的） | ❌ |
| 标记完成 / 缺勤（扣课时） | ✅ | ✅（自己的） | ❌ |
| 充值（录入线下收款） | ✅ | ✅（自己的） | ❌ |
| 课程类型管理 | ✅ | ✅（自己的一套） | ❌ |
| 统计 | ✅（全量） | ✅（自己的） | ❌ |
| **用户管理**（激活/停用/设角色） | ✅ **仅 owner** | ❌ | ❌ |
| 看课表 / 剩余课时 / 上课记录 | ✅ | ✅ | ✅**只读**（自己孩子） |
| 请假 | — | — | ✅（发起申请，非直接改课） |
| 消息中心 | ✅（自己的） | ✅（自己的） | ✅（自己的） |

**说明一期现状**：目前 owner 与 teacher 差别很小——云函数都 `requireRole(['owner','teacher'])`，owner 唯一的特权是查询不受 `ownerId` 限制（能看所有人）。**用户管理功能一期没有**，是二期新增的 owner 专属能力。

**【待定 B】** 一个要厘清的边界：teacher 之间是否互相隔离（A 老师看不到 B 老师的学员/课）？一期代码是**隔离**的（teacher 只看自己 ownerId）。二期维持隔离，还是同工作室老师共享？单工作室多老师场景，很可能希望 **owner 看全部、teacher 各管各的**（维持现状）。请确认。

### 2. 权限管理界面放哪
**【待定 C】** 放设置页的一个入口 → 独立子页。
- 一期设置页已预留"更多设置"板块（提醒设置/课时阈值/账号信息占位）。**成员管理**就作为其中一项，**仅 owner 可见**，点进去是独立子页 `pages/settings/members`。
- 为什么独立子页而不内联：成员列表有自己的列表 UI（头像/昵称/角色/状态/操作），内联进设置页会撑爆。
- 取舍：teacher 看不到这个入口（前端按 role 隐藏 + 云端 requireRole(['owner']) 双重）。

→ 推荐：**设置页里一个"成员管理"入口（owner 才显示）→ 独立子页**。

### 3. 界面化激活流程（完整交互）
现状：新用户登录即在 `users` 建档（`role:teacher, isActive:false`），进**待激活页**（🔒 + "请联系管理员激活" + 重新检测）。二期把"联系管理员+手动改库"变成界面操作：

1. **新用户**打开 → 待激活页。**新增一步**：让他填一个**姓名**（自报，例如"王老师"/"朵朵妈妈"）。因为微信已不再静默给昵称，owner 光看 openid 无法辨认谁是谁，自报名是最务实的识别手段。姓名写入 `users.displayName`。
2. **owner** 进「成员管理」→ 看到**待激活**分组（`isActive:false`），每条显示自报姓名 + 申请时间。
3. owner 对某人点**激活**（可同时选角色：teacher / student / owner）→ 云端 `setActive(true)` + 可选 `setRole`。
4. 新用户回到待激活页点**重新检测**（强制刷新登录态）→ 进入对应端首页。
5. owner 也能**停用**（isActive→false，下次进入被挡回待激活页）、**改角色**。

需要新增云函数 `usersAdmin`：`list`（全部用户，owner 专用）、`setActive`、`setRole`。均 `requireRole(['owner'])`。

**取舍**：自报姓名有被乱填的风险，但 owner 激活时人工把关即可（不认识的不激活）。比强制手机号/实名简单得多，也不碰个人主体拿不到的能力。

## 二、学生 / 家长端

### 4. 家长如何关联到学员：邀请码 / 手动绑定 / 手机号

| 方案 | 怎么走 | 优点 | 缺点 |
|---|---|---|---|
| **邀请码（推荐）** | 老师给每个学员生成一个短码，线下发给家长；家长在自己端输码 → 绑定 | 不碰隐私；线下口头/截图都能传；个人主体可用；老师可作废重生 | 码可能被转发误用（用一次性/短有效期 + 可作废来缓解） |
| 老师手动绑定 | 家长先登录出现在"待绑定"列表，老师按姓名点绑 | 老师完全掌控 | 鸡生蛋：家长没登录前老师看不到他 openid；老师要在陌生 openid 列表里靠昵称猜，容易错 |
| 手机号匹配 | 家长填手机号，匹配学员上预存的手机号 | 家长无需额外码 | 个人主体**用不了**微信手机号快速填写 API（需企业认证）；手填手机号=隐私+匹配错风险 |

**推荐邀请码**，细化：
- 码绑定到**具体学员**（不是工作室），一个学员一个码；家长输码即把自己 openid 与该学员建立关联。
- 码**可重复使用**（一个孩子父母都用同一个码分别绑），但 owner/老师能**作废/重生**。或做成**一次一绑**更安全——**【待定 D】** 你选"一码多绑（方便父母都绑）"还是"绑定需老师逐个确认"。
- 邀请码存新表 `inviteCodes`：`{ code, studentId, workspaceId, createdBy, expiresAt?, disabled, createdAt }`。

**取舍**：邀请码把分发责任交给老师（她本来就和家长有联系），换来零隐私、可控、个人主体可用。是这三者里唯一没有硬伤的。

### 5. 一家长多娃 / 一娃双亲：多对多建模
用**关联表** `guardianLinks`：
```
{ studentId, guardianOpenid, relation?（爸爸/妈妈/…）, workspaceId, createdAt }
```
- 家长的孩子们 = 按 `guardianOpenid` 查（一个家长多条 → 多个娃）。
- 一个娃的家长们 = 按 `studentId` 查（多条 → 双亲都在）。
- 天然多对多，且能挂关系、能单独解绑一方。

**对比**：一期 `users` 上已有个空的 `boundStudentIds[]` 字段，也能存"家长→孩子列表"。但反向（"这个孩子有哪些家长"）就要全表扫 users。**关联表双向都快、可加元数据、解绑干净**，更稳。
→ 推荐**关联表**；`users.boundStudentIds` 要么废弃、要么只当冗余缓存（不作为真相）。**【待定 E】** 确认用关联表。

### 6. 学生端能看什么、能操作什么
- **只读**：自己孩子的**课表**（将上 + 已上）、**剩余课时**、**上课记录**（出勤/缺勤历史）、**消息**（与自己孩子相关的提醒）。
- **唯一可操作：请假**。且是"**发起申请**"，不是直接改课——家长点某节课"请假" → 生成一条待处理申请 → **老师收到消息** → 老师决定标缺勤/改期。老师始终掌控课表真相，家长不能直接动数据。
- 其它一律不可：不能建/改课、不能改学员、不能充值。

请假存新表 `leaveRequests`：`{ sessionId, studentId, guardianOpenid, reason?, status: pending/approved/rejected, workspaceId, createdAt }`。

**取舍**：把家长能力压到"只读 + 请假申请"，既满足家长核心诉求（看课、看余额、请假），又不给课表真相引入不可控写入。请假走"申请-审批"而非直接生效，是关键的一道闸。**【待定 F】** 是否本期就做请假？还是家长端先只读，请假二期下半程再加？

### 7. 充值（个人主体不能在线支付）
- **家长端不做在线支付**（个人主体没这能力）。
- 课时不足时：家长端**提醒 + 引导线下联系老师**（显示"课时不足，请联系老师续费"，可放老师微信/二维码，**【待定 G】** 要不要放联系方式、放什么）。
- **充值记录家长可见（只读）**：老师线下收款后，用现有充值功能录入 → 家长端能看到自己孩子的**课时流水**（"充值 10 课时，余额 12""扣 1 课时，剩余 9"），和消息中心里的充值/扣课时消息。
- 即：**收款线下、录入在老师端、结果家长可查**。透明但不经手钱。

**取舍**：既守住合规（不碰在线支付），又让家长对余额变动心里有数（可见流水），避免"我明明充了怎么没加"的纠纷。

## 三、架构

### 8. 除 workspaceId，还要现在埋哪些字段
先分清两类：**改老表加字段（要回填，趁数据少现在埋）** vs **新建表（无存量，二期建即可，不用现在埋）**。

**要现在埋的（改老表）**：
- **只有 `workspaceId`**（见第〇节）。这是唯一"晚埋要付迁移代价"的。
- 可选：给所有表加一个 `schemaVersion`（数字），将来结构迁移时好按版本处理。成本极低，**【待定 H】** 加不加。
- `users`：二期要加 `displayName`（自报名，见第 3 点）。但 users 无存量隔离问题，新字段默认空即可，不算"必须现在埋"。

**不用现在埋的（二期新建表，自带字段）**：
- `guardianLinks`（家长-学员多对多，第 5 点）
- `inviteCodes`（邀请码，第 4 点）
- `leaveRequests`（请假，第 6 点）
- 这些都没有一期存量数据要回填（现在还没有家长），二期建表即可。

**关于"students 加 boundOpenids"**：家长关联我推荐走**关联表**而非在 students 上挂数组（见第 5 点），所以 **students 不需要为此加字段**。若你最终选"数组挂 students"，那才需要现在给 students 埋 `guardianOpenids: []`（回填空数组，成本很低）。取决于第 5 点【待定 E】。

→ **结论：现在必须埋的就是 `workspaceId`。** 其余要么是新表（无需回填），要么取决于建模选择。

### 9. 学生端 vs 老师端：角色分流 vs 入口页
**推荐：同一小程序，登录后按 role 自动分流，不做入口页。**
- 登录已返回 role → 首页 route 时：owner/teacher 进老师端首页，student 进家长端首页。无需用户选。
- **入口页（"我是老师/我是家长"）什么时候才需要**：当**同一个微信号同时是老师又是家长**（比如某老师的孩子也在这上课），一个 openid 两种身份，系统不知该进哪端。
  - 这是**少数边缘情况**。为它给**所有人**加一步选择页，不划算。
  - 更好的处理：这类人给一个"**切换身份**"入口（放个人/设置区），默认进主身份，需要时切。而不是开屏拦所有人。
- **取舍**：入口页 = 给 100% 用户加摩擦去服务 <5% 的双身份用户。推荐**不做入口页**，用"角色自动分流 + 双身份者可切换"。**【待定 I】** 确认。

### 10. 一期能力：学生端复用哪些、隔离哪些

| 一期能力 | 学生端怎么处理 |
|---|---|
| **冲突检测** | **完全隔离**。冲突检测是排课时的老师侧逻辑；家长不排课，用不到。 |
| **课程类型** | **复用快照，隔离管理**。课程名早已快照在每节课上（`courseTypeName`），家长看课直接读快照，**不需要**课程类型管理界面，也不查 courseTypes 表。 |
| **消息中心** | **复用基础设施，扩展收件人**。现在 `notifications.ownerId` = 老师 openid（收件人就是老师）。二期要能给**家长**发（孩子的课前提醒、课时不足、请假结果）。 |

**消息中心的关键改造（第 10 点重点）**：
- 现在通知的"收件人"语义借用了 `ownerId`（老师）。二期通知要发给家长，收件人不再只是"owner"。
- 方案：把收件人语义**显式化**——通知加/改成 `recipientOpenid`（收件人 openid），`ownerId` 语义不再够用。列表/未读/已读都按 `recipientOpenid = 当前用户` 查。老师的通知 recipient=老师 openid，家长的 recipient=家长 openid。
- **写入端扇出**：像"课前提醒"这种，二期可能要**同时发给老师和该课每个学员的家长**——一个事件写多条通知（每个收件人一条）。课时不足则发给老师 +（可选）家长。
- 复用：`notifications` 集合、`notifications` 云函数（list/unreadCount/markRead）、清理逻辑、消息中心页 UI，全部复用，只是多了"按收件人"和"扇出多收件人"。
- **【待定 J】** 收件人字段：改造现有 `ownerId`→`recipientOpenid`（需回填存量：把老师的通知 recipient 设为 ownerId），还是加新字段 `recipientOpenid` 并存两套。推荐**改造+回填**（一期通知量本就会被 90 天清理规则慢慢清掉，回填成本低）。
- **【待定 K】** 家长要不要收"课前提醒"订阅消息？还是家长端只在应用内消息中心看、不占用订阅额度？（家长订阅又是一套授权与额度问题。）建议：**家长端先只做应用内消息，不发订阅消息**，避免又一轮订阅授权复杂度；跑通后再评估。
