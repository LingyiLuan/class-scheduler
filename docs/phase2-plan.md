# 二期设计文档（定稿）

本文件为二期最终决策集，覆盖此前所有讨论。下方 A–K 决策、学员归属模型、workspaceId 预留、开发顺序、回滚方案、加载诊断均为定稿。开发按此执行。

---

## 定位

- **单工作室工具**，只服务这一家；不做多工作室 SaaS。
- 朋友是这家的 **owner / admin**。
- **轻度预留平台化**：核心表加 `workspaceId`，当前全填同一固定值 `ws_main`，查询暂不强制过滤。

## 学员归属（核心数据模型）

**学员归工作室，不归老师。** 这是与一期最大的行为变化。

### 共享（workspaceId 维度，所有老师可操作）
- `students`（学员信息、课时余额）：任何老师都能建、排课、充值、查余额
- `creditLogs`（课时流水）：**共享账户**，谁扣都从同一余额扣
- `classSessions`（课表）：冲突检测按 workspaceId **全局**查
- `courseTypes`、`packages`

### 私有（ownerId 维度）
- 首页默认视图：只显示"我排的课"（但建课查冲突是全局的）
- 统计业绩：一节课的消耗算 `session.ownerId`（谁上算谁的）
- 消息中心：各收各的

### 规则
1. 一个学员可被多个老师排课，**共用一个课时账户**。
2. 扣课时：从学员同一余额扣，session 记在上课老师名下。
3. 业绩：消耗课时按 `session.ownerId` 归属。
4. 充值：任何老师都能充，充进工作室账户，**不算个人业绩**。
5. 学员列表：所有老师看到**同一批工作室学员**，不再按 ownerId 过滤。
6. 学员删除：有课程引用的学员**不能硬删，只能停用**（防止 A 删掉 B 正在教的学员），参照课程类型删除策略。

## workspaceId 预留细节

- **加字段的表**：`students` / `classSessions` / `recurrences` / `packages` / `creditLogs` / `courseTypes` / `notifications`
- **不加**：`users`（用户跨工作室，将来用成员关系表达）
- **存量回填**：一次性迁移把缺字段的记录补成 `ws_main`，幂等
- **新写入**：云端在写入处统一注入 `workspaceId`
- **查询**：一期不加过滤，**只埋不用**
- `schemaVersion` 同一次迁移一起写（初值如 `1`）

## 现实：这家有 2-3 个老师（共享课表本期做）

**前提：这家现在 2-3 个老师**（不是单老师）。2-3 人同时排课，会真实发生**跨老师给同一学生撞时间**。除了后台冲突检测（硬拦截），老师还需要**看到彼此的课表来主动协调**。因此：

- **共享课表 UI 本期就做**（不延后），紧接冲突检测（第 3.5 步）：
  - 老师能在课表里看到**全工作室的课**
  - 视觉区分**"我的课 / 别人的课"**，别人的课显示是**哪位老师**的
  - **别人的课只读**，不能改/删（改删仍受 ownerId 限制）
- **加载优化必须在 workspaceId 改造之前做**（第 0.5 步）：三个老师共享学员后，首页/学员页 N 次 `getBalance` 的 N 变成**全工作室学员总数**（可能是单老师的 3 倍）。带着已知 N+1 去做**放大它**的改造 = 叠加风险。先消 N+1，再共享。

### 前置：把 2-3 个老师引导进来（不需要第 5 步 UI）
共享功能要有真实多老师数据，需先让另外两位老师成为**已激活的 teacher**。这**不依赖第 5 步**：
1. 每位老师**打开一次小程序** → login 自动建档（`role:teacher, isActive:false`）→ 停在待激活页。
2. 你在控制台 `users` 集合把他们的记录 `isActive` 改为 `true`（role 保持 teacher）。
3. 他们回待激活页点「重新检测」即进入，可开始排课。

第 5 步（用户管理 UI）是把这个手动过程**产品化**供将来自助增减用，**不是共享功能的技术前置**。是否把第 5 步提前由你定；本文档默认第 5 步在原位、靠上述手动引导先解锁多老师数据。

## 权限模型

### 角色能力矩阵（owner=admin / teacher / student）

| 能力 | owner | teacher | student（家长） |
|---|---|---|---|
| 看数据范围 | 全工作室 | **工作室共享学员**（B：可见共享）；自己排的课默认视图 | 仅关联孩子，只读 |
| 建/改/删学员 | ✅ | ✅（工作室共享，删除只能停用） | ❌ |
| 排课 / 改期 / 取消 | ✅ | ✅ 建课全局查冲突；**只能改删自己 ownerId 的课** | ❌ |
| 标记完成/缺勤（扣课时） | ✅ | ✅（记在自己名下） | ❌ |
| 充值（录入线下收款） | ✅ | ✅（充进工作室账户，不算个人业绩） | ❌ |
| 课程类型管理 | ✅ | ✅ | ❌ |
| 统计 | ✅ 全量 | ✅ 仅自己 ownerId 业绩 | ❌ |
| **用户管理**（激活/停用/设角色） | ✅ **仅 owner** | ❌ | ❌ |
| 看课表/余额/上课记录 | ✅ | ✅ | ✅ 只读（自己孩子） |
| 消息中心 | ✅ 自己 | ✅ 自己 | ✅ 自己 |

### 成员管理界面（决策 C）
- 设置页内一个"成员管理"入口，**仅 owner 可见**（前端按 role 隐藏 + 云端 `requireRole(['owner'])` 双重）。
- 点进独立子页 `pages/settings/members`：用户列表（自报姓名/角色/状态/操作），支持激活、停用、改角色。
- 需新增云函数 `usersAdmin`：`list` / `setActive` / `setRole`，均 owner 专用。

### 界面化激活流程
1. 新用户登录即建档（`role:teacher, isActive:false`）→ 待激活页。**新增**：让他填**自报姓名**（写入 `users.displayName`；微信不再给昵称，owner 靠此辨识）。
2. owner 进「成员管理」→ 待激活分组（`isActive:false`），显示自报姓名 + 申请时间。
3. owner 点激活（可同时选角色 teacher/student/owner）→ `setActive(true)` (+ 可选 `setRole`)。
4. 新用户回待激活页点「重新检测」（强制刷新登录态）→ 进入对应端。
5. owner 可停用、改角色。

## 家长端

### 关联方式（决策 D：邀请码，一码多绑）
- 老师给每个学员生成一个短码（绑定到**具体学员**），线下发给家长。
- 家长在自己端输码 → 把自己 openid 与该学员建立关联（写 `guardianLinks`）。
- **一码多绑**：父母、老人可用同一码分别绑定。owner/老师可**作废/重生**兜底。
- `inviteCodes` 表：`{ code, studentId, workspaceId, createdBy, disabled, createdAt }`。

### 多对多建模（决策 E：关联表）
`guardianLinks`：`{ studentId, guardianOpenid, relation?, workspaceId, createdAt }`
- 家长的孩子们 = 按 `guardianOpenid` 查；一个娃的家长们 = 按 `studentId` 查。
- 天然多对多，可挂关系、可单独解绑。`users.boundStudentIds` 废弃。

### 能看/能做（决策 F：本期只读）
- **只读**：自己孩子的课表（将上+已上）、剩余课时、上课记录、消息。
- **本期不做请假**。家长端纯只读；请假（申请-审批模式）延后。

### 充值（决策 G/K）
- 家长端不做在线支付（个人主体不能）。
- 课时不足页：显示"课时不足，请联系老师续费" + **老师微信二维码**（老师在设置里上传二维码图片）。
- 收款线下、老师录入（现有充值功能）、家长可见**课时流水**（充值/扣课时，只读）。
- 家长端提醒**只走应用内消息中心，不发订阅消息**（决策 K）。

## 架构

### 现在埋哪些字段
- **必须现在埋（改老表、要回填）**：`workspaceId`、`schemaVersion`（7 张表）+ `students.aliasCn`（班级设计第 9 点，随同一次迁移）。这是"晚埋要付停机迁移代价"的部分。
- **不用现在埋（二期新建表，自带字段）**：`classes`、`classEnrollments`（班级，第 3.7 步；空集合可在第 1 步顺手建好）、`guardianLinks`、`inviteCodes`、`leaveRequests`（请假延后）。无存量数据，建表即可。
- `workspaceConfig`（工作室欢迎语）已在一期末（反馈2）引入。
- `users` 二期加 `displayName`（一期末已加字段+设置接口，成员管理复用）、家长的 `role='student'`；无隔离存量问题。
- 学员-家长、学员-班级都走关联表，**students 只加 `aliasCn`，不加 boundOpenids / classId**。

### 建课时 session 新增快照字段（第 3.7 步）
- `classSessions` 选班建课写 `classId`（可空）+ `className`（班级名快照）；成员仍走 `studentIds` 快照。

### 端分流（决策 I）
- 同一小程序，登录后按 `role` 自动分流：owner/teacher → 老师端首页；student → 家长端首页。
- 不做入口页。双身份（既老师又家长）给"切换身份"入口（设置区），默认进主身份。

### 一期能力：复用/隔离
| 一期能力 | 学生端 |
|---|---|
| 冲突检测 | **完全隔离**（家长不排课） |
| 课程类型 | **复用快照，隔离管理**（家长看课读 `courseTypeName` 快照，不碰 courseTypes 表） |
| 消息中心 | **复用基建，扩展收件人**：通知 `ownerId`→`recipientOpenid`（决策 J）；写入端对"课前提醒"等事件**扇出多收件人**（老师 + 该课学员的家长）；`notifications` 集合/云函数/页 UI 全复用 |

## 开发顺序（按"能独立验证、风险低→高"）

> 确认基础：先做**第 0 步加载诊断**（见下节），再进二期。以下每步做完即可独立验证，验证通过再进下一步；不叠加半成品。

**前置 · 引导 2-3 个老师进来**（手动，非编码）
- 每位老师开一次 app 建档 → 你在控制台把 `isActive` 改 true（role=teacher）。为后续共享/冲突步骤提供真实多老师数据。详见"现实"章节。

**第 0 步 · 加载慢诊断**（二期开工前，见最后一节）
- 独立验证：拿到关键路径分段耗时（冷启动/查询/网络），确认瓶颈，产出结论。**先诊断，不优化。**

**第 0.5 步 · 加载优化（必须在 workspaceId 改造之前）**（中风险，改余额查询路径）
- 新增批量余额接口 `credits.balances({studentIds})`（一次聚合多学员余额）；首页与学员页从 `Promise.all(map(getBalance))` 改为**一次**批量调用；给 `creditLogs.studentId` 建索引。
- **为什么在 workspaceId 前**：共享学员后 N（getBalance 次数）会膨胀到全工作室学员数；先消 N+1 再放大，避免叠加风险。
- 独立验证：首页/学员页网络请求从 N 次 → 1 次；余额数值与改造前逐一致；计时日志显示该段耗时显著下降。

**第 1 步 · workspaceId + schemaVersion 预留 + 回填**（低风险，纯加字段）
- 迁移把 7 张表缺字段的记录补 `ws_main` / `schemaVersion:1`；云端写入处统一注入。查询不变。
- **同批一起做**：给 `students` 加 `aliasCn`（中文名，回填 `''`，见班级设计第 9 点）；创建空集合 `classes`、`classEnrollments`（无存量、无需回填，供第 3.7 步用）。
- 独立验证：控制台查每张表所有记录都有 workspaceId/schemaVersion；students 有 aliasCn；空集合已建；新建学员/课程后字段存在；**app 行为与改造前完全一致**（因为不过滤）。

**第 2 步 · 学员查询改 workspaceId（共享学员）**（中风险，改查询 + 删除保护）
- `students.list` 等去掉 `ownerId` 过滤，改按 `workspaceId`；学员删除改"有引用只能停用"。
- 独立验证：owner 列表改造前后一致（owner 本就看全部）；**teacher 列表改造后看到全工作室学员**（这是预期的行为变化）。用真实 2-3 老师数据核对：任一老师建的学员其他老师都能看到；被课程引用的学员无法硬删（只能停用）。

**第 3 步 · 冲突检测改 workspaceId（全局冲突）**（中风险，改查询）
- `findConflicts` 与 recurrences 冲突查询从 `ownerId` 改 `workspaceId`。
- 独立验证：单老师下冲突结果不变。跨老师验证：造两条不同 ownerId、同时段、含同一学员的课，确认硬冲突跨老师触发；不同学员同时段仍软提示。

**第 3.5 步 · 共享课表 UI**（中风险，展示层；紧接冲突检测）
- 课表/首页拉取范围从"我的课"扩到**全工作室的课**（按 workspaceId）；每节课带上 `ownerId`（及老师显示名）。
- 视觉区分**我的课 / 别人的课**，别人的课标注**是哪位老师**的；别人的课**只读**（点开不显示改/删/完成等操作，或操作按钮按 `ownerId !== 我` 隐藏）。
- 首页默认仍只显示"我的课"，课表页可切到"全部/我的"（**【待定】** 默认视图与切换粒度，实现时定）。
- 独立验证：两位老师各排几节课 → 在 A 的课表里能看到 B 的课、标着 B 的名字、只读（改删入口不出现）；A 自己的课仍可改删；跨老师撞同一学生时冲突拦截（第 3 步）叠加此可见性，老师能主动错开。

**第 3.7 步 · 班级**（中风险，新表 + 排课体验；紧接共享课表，见 `docs/classes-plan.md`）
- `classes` 增删改停用（软删同课程类型）；`classEnrollments` 桥表（学生-班多对多，界面默认单班）；学员编辑班级单选下拉。
- 建课新增"选班"→ 查桥表预填 `studentIds` + 快照 `className`；选人/列表显示 `aliasCn`「英文名（中文名）」；20+ 人点名默认全到只勾缺席；班级课可单独移除某生（改本节 studentIds，不动归属）。
- 依赖第 2 步（学员已工作室共享），班级才有干净地基。
- 独立验证：建个班、拉几个学生入班 → 选该班建课，studentIds 自动填满、课上带班级名；改班级成员后，已排的历史课名单不变（快照）；被引用的班无法硬删只能停用；同名学生在选人列表显示中文名可区分。

**第 4 步 · 通知 recipientOpenid 改造 + 回填**（高风险，动存量+多处读写）
- 用"加新字段、留旧字段、切读、后删旧"的安全模式（见回滚方案）。
- 独立验证：存量通知仍能在消息中心显示（recipient 已回填）；新事件写入的通知正常显示；未读计数正确；老师端一切如常。

**第 5 步 · 用户管理 / 激活界面**（低-中风险，新增功能，不动存量）
- `usersAdmin` 云函数 + 成员管理子页 + 待激活页自报姓名。
- 独立验证：用第二个微信号登录 → 出现在待激活列表（带姓名）→ owner 激活 → 第二号进入 app；停用后被挡回。

**第 6 步 · 邀请码 + 家长关联**（新增表与流程，不动存量）
- `inviteCodes` + `guardianLinks` + 生成码/输码/作废。
- 独立验证：老师给某学员生成码 → 家长号输码 → `guardianLinks` 生成 → 家长号能查到该孩子；作废后码失效；一码多绑可绑第二个家长。

**第 7 步 · 家长只读端**（新增端，按 role 分流）
- role=student 分流 + 只读页（课表/余额/记录/消息）。
- 独立验证：家长号只看到自己孩子的数据，全部只读，改不动任何东西；看不到别的孩子。

**第 8 步 · 家长消息（应用内）**（复用消息基建，扩展扇出）
- 相关事件对家长收件人扇出应用内通知（不发订阅消息）。
- 独立验证：标记完成某节课 → 该课孩子的家长号消息中心出现对应通知；老师侧通知不受影响。

## 回滚方案（三个高风险改造，先写清再动手）

**通用前置**：每个动存量的迁移**执行前先导出对应集合**（控制台数据库导出，或先跑一次只读的"影响面统计"）。迁移**幂等**、**只增不删**。一步一验证，不连做。

### R1 · workspaceId / schemaVersion 回填
- **本质无损**：迁移只**新增字段**，不改不删任何现有字段。且第 1 步查询**不使用**该字段，写错值也零运行时影响。
- **回滚**：
  1. 若值写错 → 重跑一次修正迁移（全置 `ws_main`），因为幂等、只覆盖这俩字段，不碰业务数据。
  2. 若要彻底撤回 → 重新部署改造前的云函数版本（新写入不再注入该字段）；已写入的多余字段**惰性无害**，老代码不读它，可留可后删。
- **最坏情况恢复**：用执行前导出的集合快照重新导入。因为本步不删数据，几乎不会走到这一步。

### R2 · 学员查询改造（共享学员）
- **本质是代码改动**（查询条件 `ownerId`→`workspaceId`），本步**不含数据迁移**（字段已在 R1 备好）。
- **回滚**：`git revert` 该提交 + **重新部署改造前的 `students`（及相关）云函数版本** → 立即回到"按 ownerId 过滤"的一期行为。数据完全未动。
- **验证闸**：改造前后各记录一次列表返回的学员 id 集合——**owner** 前后应一致（本就看全部）；**teacher** 应从"仅自己的"扩为"工作室全集"（这是预期变化，核对正好是 workspace 全集，不多不少）。异常就别上，先查。
- **删除保护**若误伤（本可删的删不了）→ 同样 revert 代码即可，无数据风险。

### R3 · recipientOpenid 回填（最危险，动存量 + 多处读写）
- **强制用安全模式，不做原地改名**：
  1. **加新字段** `recipientOpenid`，**保留 `ownerId` 不动**。
  2. 迁移把存量通知 `recipientOpenid = ownerId`（复制，不删 ownerId）。
  3. 读/写代码切到 `recipientOpenid`；`ownerId` 作为**死副本**保留一个版本周期。
  4. 观察稳定后，**下一个独立迁移**才删 `ownerId`（那步同样先导出、幂等）。
- **回滚**：因为 `ownerId` 一直还在 → **重新部署读 `ownerId` 的旧代码** 即恢复；`recipientOpenid` 多余字段惰性无害。
- **最坏情况恢复**：迁移前导出 `notifications`，异常时重新导入。
- **要点**：改造 ≠ 直接改名删旧。任何"重命名字段"都拆成"加新→回填→切读→(隔期)删旧"四步，每步可回退。

## 加载慢诊断（二期开工前先做，先诊断不优化）

### 静态结论（已从代码确认）

1. **学员列表未真正分页**：`students.list` 是 `where(...).orderBy('createdAt','desc').limit(100).get()`——**硬上限 100、无 skip 分页**。<100 人时一次查询返回；**≥100 人会被静默截断**（这既是性能、也是潜在正确性问题）。
2. **余额是实时聚合，非存字段**：`credits.getBalance`（及 sessions/packages 内的 `computeBalance`）都是对该学员**全部 `creditLogs` 聚合求和 `delta`**。**没有余额快照字段**——每次读余额都扫一遍该学员流水。
3. **确认存在 N+1，且在两处**：
   - 首页 `loadData`：`listSessions` → `listStudents` → **`Promise.all(students.map(getBalance))`** → `unreadCount`。
   - 学员列表页：同样 **`Promise.all(list.map(getBalance))`**。
   - 即：**每进一次首页/学员页，就有 N 次 `credits.getBalance` 云调用**（N=学员数，最多 100）。**二期学员改为工作室共享后，N 会更大**（看到全部学员而非单老师的），N+1 更严重。
4. **索引**：代码不建索引（手动在控制台建）。关键疑点：`creditLogs.studentId`（聚合 match 字段）大概率**没建索引** → 每次 getBalance 全表扫 creditLogs；`students.ownerId/workspaceId`、`classSessions` 的 `ownerId+startTime` 同样待确认。
5. **冷启动链路**：`app.ts` 仅 `cloud.init`（轻）；但首屏 route 前要先 `ensureLogin`（login 云函数）才分流。**首个云调用叠加云函数冷启动开销**。

### 初步判断（待计时确认）
2–3 秒大概率由 **N 次并发 `getBalance` 云调用**主导：每次含"云函数调用开销 +（无索引时）creditLogs 聚合扫描"，N 个并发还受云函数实例并发上限影响；再叠加首个调用的冷启动。**这是最大嫌疑，但要用计时日志坐实，不能凭猜优化。**

### 计时instrumentation 方案（下一步做，非本轮）
在关键路径加**开发环境计时日志**（正式包不输出）：
- `api.ts`：每次 `callFunction` 记 `name + 耗时ms`，并标记"本次启动后的首个调用"（区分冷启动）。→ 得到每个云调用的网络+云端总耗时。
- 首页 / 学员页：记 `route 开始 → 数据就绪` 总耗时，及各子步骤（login / listSessions / listStudents / 全部余额 / unread）分段。→ 区分是"某一步慢"还是"N 次余额慢"。
- 云端 `credits.getBalance`、`students.list`：`console.time` 包住聚合/查询部分。→ 把"查询耗时"从"调用开销"里剥出来。
- 跑一版带计时的**真机**构建，采集客户端 console + 云开发日志，回答：冷/热首屏各多久、余额 N 次占比、单次聚合耗时、是否缺索引。

### 待诊断产出回答的问题
- 学员列表是否分页 → **否，硬 limit 100**（已确认，需改真分页或批量）。
- 剩余课时实时聚合还是存字段 → **实时聚合，无快照**（已确认）。
- 有没有 N+1 → **有，首页与学员页各一处**（已确认）。
- 索引情况 → 需登控制台核对（重点 `creditLogs.studentId`）。

### 诊断结论与优化进度（实测：N+1 主因）
实测 45 学员：首页热启动 8s，其中 `balances×45` 占 57%、学员页占 89%；单次干净 getBalance 717ms，45 次并发却 4.5-6s（受云函数实例数限制，非线性）。冷启动是次要项。
1. ✅ **批量余额接口** `credits.balances({studentIds})`（第 0.5 步已做）：一次 `match(in).group(studentId).sum` 返回映射；首页/学员页 N 次 → 1 次。
2. ⬜ `creditLogs.studentId` 索引（Ascending / Non-unique，手动建）——批量后那一次聚合更依赖它。
3. ⬜ **静默截断三处（phase2 待办，同类问题）**：
   - `students.list` 硬 `limit(100)` 无 skip：≥100 学员静默丢弃 → 改真分页（skip/游标）。
   - `credits.balances` 聚合已加 `.limit(学员数)` 修了默认 20 组截断；但**云函数端聚合单次上限 1000**，学员 >1000 时一次聚合返回不全 → 需**分批聚合**（studentIds 切块 ≤1000 各聚合再合并）+ 注意 `_.in` 超大数组。当前 25-45 人远未到，不做。
   - 通用规则：凡 `aggregate().group(by 字段)` 必加 `.limit()`；`.get()` 注意 100/1000 上限。
4. ⬜ 次要：首屏 4 个云函数**串行**（listSessions→listStudents→balances→unread，耗时相加）。可**并发化**（无依赖的并发）或**合并为一个 home 函数**（减少冷启动次数）。批量做完若仍慢，这是下一杠杆。
5. 谨慎的长期项：余额快照字段（违背"creditLogs 唯一真相"，需事务维护，非必要不做）。
