# 班级 功能设计（待确认，未写代码）

来源：一个老师二十多个学生，英文名常重复，希望**以班级为单位建课**。逐条给方案与取舍。

---

## 1. 班级和学员是什么关系

### 方案对比

| 方案 | 建模 | 优点 | 缺点 |
|---|---|---|---|
| **一生一班（推荐 MVP）** | `students.classId`（可空） | 简单；一次迁移/一个字段；选班即选一组人 | 一个学生只能属一个班；跨班（正课+考前班）要靠"临时选人"补 |
| 多对多 | 关联表 `classMembers{studentId,classId}` | 一个学生可在多个班 | UI 与心智更复杂；MVP 用不满 |

### 推荐
**一生一班（`students.classId`）作 v1**。
- 语言工作室里，孩子通常有一个"主班"（自己的级别组）。跨班的**补课**本来就是**单次课临时选人**，一期"多选学员建课"已经能覆盖，不必为它上多对多。
- 升级路径留着：真出现大量跨班需求，再从 `classId` 平滑升到 `classMembers` 关联表（把每个学生现有 classId 迁一条关联记录即可，无损）。

**取舍**：一对多牺牲"一生多班"的灵活，换来简单直接；而"一生多班"的真实场景（补课）已有单次多选兜底，损失很小。

**【待定 1】** 一生一班 / 多对多。推荐一生一班。

## 2. "以班级为单位排课"具体指什么

### 方案
**选一个班 = 给班里当前所有（在读）学生排同一节课。** 落到数据就是：一节 `classSession`，其 `studentIds` = 该班当前成员快照。

- **班级只是"选人快捷方式"**：建课时选班 → 自动把该班成员填进 `studentIds`。生成的仍是一期那种**多学员单节课**，模型完全复用（sessions 早就有 `studentIds[]` + 逐人 `attendance`）。
- **存快照**：session 存 `classId`（便于显示/筛选"某班的课"）**+ `studentIds` 成员快照**（建课时的名单）。之后班级增减人，**不影响已建的课**（与课程类型快照同理）。
- **课时怎么扣**：**每个学生各扣自己的**。一期"标记完成"已是**逐个出席学员从各自账户扣 1**（`creditLogs` 按 studentId）。班级课就是 studentIds 多几个人而已，扣课时逻辑一行不用改：谁出席扣谁的、谁请假不扣。
- **点名 UI**：小班课的出勤勾选要能容纳 20+ 人（现在的 attendance sheet 需支持滚动/全选/批量，见【待定 3】）。

### 取舍
把"班级课"设计成"**班级=预填 studentIds 的语法糖**"，而不是新的一等公民实体，最大化复用一期的扣课时/出勤/冲突检测，风险最低。代价：班级本身能力较薄（就是个名单 + 归类），但正好匹配"我要快速给这组人排课"的诉求。

**【待定 2】** 认可"班级课 = 多学员单节课 + classId 快照 + 逐人扣课时"？
**【待定 3】** 20+ 人点名交互：默认全到、只标缺勤的（省点击）？还是逐个勾？建议**默认全到、只勾缺席**。

## 3. 英文名重复的区分

班级能**缩小范围**（先选班，只看这个班的名单），但同名仍可能同班。补两招：

- **加一个显示用的区分字段**：学员加 `aliasCn`（中文名）或复用现有 `note`，在**所有选人/列表处**显示成「Amy（王小美）」。低成本、直接解决"两个 Amy"。
- **选人列表带副信息**：选学员时同时显示 班级 / 级别标签 / 中文名，靠上下文区分。

### 推荐
**班级分组 + 中文名/备注双管齐下**：
- 建课选班后，名单里每人显示「英文名（中文名）」。
- students 增一个短字段（`aliasCn`，≤10 字）专门用于并列显示；不与 `note`（自由备注）混用，保证展示稳定。

**【待定 4】** 加专用 `aliasCn` 字段（推荐） / 复用现有 `note` 显示。

## 4. 与二期"学员归工作室"的交集 —— 要合并设计

**有交集，且必须一起设计。** 理由：

1. **班级也归工作室，不归老师**：一个班（如"剑桥 KET 周六班"）是工作室资产，**任何老师都能用它排课**。所以 `classes` 表从一开始就带 **`workspaceId`**（与二期同款预留），归属走 workspace 而非某老师。
2. **班级建在"共享学员"之上**：班级成员是学员，而学员二期正变成**工作室共享**。所以班级功能应**排在"学员查询改 workspaceId（二期第 2 步）"之后**——共享学员就位了，班级才有干净的地基（任何老师都能把任何工作室学员编进班）。
3. **字段预留一并做**：`students.classId`、`students.aliasCn` 这两个新字段，跟二期第 1 步的 `workspaceId/schemaVersion` **同一次迁移**加上（默认 `classId:null`、`aliasCn:''`），省一次全表迁移。

### 新增表 / 字段一览
- 新表 `classes`：`{ name, workspaceId, isActive, sortOrder, createdAt }`（软删同课程类型：被学员引用的班只能停用不能硬删）。
- `students` 加：`classId`（可空）、`aliasCn`（中文名，可空）。
- `classSessions` 建课时写：`classId`（来源班，可空）+ 沿用 `studentIds` 快照。

### 建议排期（挂在二期开发顺序里）
- 二期**第 1 步迁移**：workspaceId/schemaVersion **+ students.classId/aliasCn** 一起埋。
- 二期**第 2 步（学员共享）之后**，插入 **"班级"步**：`classes` 增删改停用 + 学员分班 + 建课选班 + 选人处显示中文名 + 20 人点名 UI。位置大约在第 3.5 步（共享课表 UI）前后——都属老师端排课体验，可相邻做。
- 家长端（第 7 步）里，家长看到孩子的课自然带班级信息，无需额外设计。

**【待定 5】** 认可"班级并入二期、字段随第 1 步迁移一起埋、功能排在学员共享之后"？
