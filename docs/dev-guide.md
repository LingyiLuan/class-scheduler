# 开发规范

本文件面向本项目的所有开发工作，包括人工与 AI 辅助编码。产品背景与功能范围见 `plan.md`。

---

## 一、项目定位

英语工作室的排课与学员管理工具，微信小程序。一期只做老师端，单用户使用，不涉及支付与学生端。

技术选型的核心约束是交付周期短（14 天），因此优先选择开箱即用的方案，避免自建后端与过度抽象。

---

## 二、技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 框架 | Taro 3 + React + TypeScript | 保留后续编译 H5 与 App 的能力 |
| UI | NutUI React (Taro 版) | 不引入第二套组件库 |
| 后端 | Supabase | 数据库、接口、定时任务一体 |
| 数据库 | PostgreSQL | Supabase 托管 |
| 日期处理 | dayjs | 统一使用，不混用原生 Date 运算 |
| 包管理 | pnpm | |

不引入 Redux 或其他全局状态库。一期数据规模小，React 内置状态加 Supabase 查询足够。

---

## 三、目录结构

```
src/
  pages/          页面，每页一个目录
  components/     跨页面复用组件
  services/       Supabase 数据访问层，每张表一个文件
  hooks/          自定义 hooks
  utils/          纯函数工具
  types/          TypeScript 类型定义
  constants/      枚举与常量
docs/
  plan.md              项目方案
  dev-guide.md         本文件
  scheduling-rules.md  排课规则定义
  schema.sql           数据库建表语句
```

页面组件不直接调用 Supabase 客户端，一律通过 `services/` 层。这样后续如果更换后端，改动收敛在一个目录内。

## 四、数据设计原则

### 课时记账

课时余额由 `credit_logs` 表的 `delta` 字段累加得出，不设可直接修改的余额字段。

任何课时变动都必须写入一条流水，包含变动原因与关联的课程 ID。这样出现争议时可以完整回溯每一次扣减。

余额通过数据库函数 `student_balance(student_id)` 查询。如果性能出现问题，再考虑加缓存字段，但缓存字段只能由流水触发更新，不可手工修改。

### 状态枚举

课程状态：`scheduled` / `completed` / `absent` / `cancelled`

流水原因：`purchase` / `attend` / `absent` / `manual`

所有枚举值集中定义在 `src/constants/`，不在业务代码里写字符串字面量。

### 时间处理

数据库统一存 `timestamptz`，前端统一用 dayjs 处理。循环课的规则存 `weekday` 与 `time_of_day`，具体课程实例存完整时间戳。

## 五、排课模块

这是工作量与风险最集中的模块，实现前必须先完成 `docs/scheduling-rules.md`，明确以下问题：

- 循环课一次生成多少周，是一次性生成实例还是按需滚动生成
- 修改单节课时是否影响整个循环系列
- 删除循环规则时已生成的课程如何处理
- 冲突检测的判定标准
- 状态流转中哪些操作可逆，各自对课时的影响

规则未确认前不开始编码。

## 六、代码约定

### 命名

- 组件文件 PascalCase，其余 camelCase
- 类型定义与表名对应，如 `Student`、`ClassSession`
- 布尔变量以 `is` / `has` / `can` 开头

### 注释

只在两种情况写注释：解释为什么这样做，以及标注临时方案与已知限制。不写复述代码行为的注释。

### 提交

每个功能模块完成后提交一次，message 使用中文：

```
feat: 学员管理页面
fix: 修复循环课跨月生成错误
refactor: 抽取课时计算逻辑
docs: 补充排课规则说明
```

不提交 `.env`、`node_modules`、`dist`。

---

## 七、开发顺序

按以下顺序推进，每个模块完成后确认再继续，避免一次性铺开：

1. 学员管理（列表、新增、编辑、详情）
2. 课时充值（新增课包并写入流水）
3. 排课日历（周视图、单次课、循环课、冲突检测）
4. 上课记录（完成与缺勤，扣减课时）
5. 首页（今日课程、本周统计、课时预警）
6. 订阅消息提醒

## 八、界面文案

一期使用工具类定位，界面文案统一采用「课表」「学员」「课时」「记录」等表述。

不出现「报名」「购买」「课程销售」「培训机构」等指向教育服务与交易的措辞，以确保工具类目审核顺利。二期升级企业主体后不受此限制。

## 十、待确认事项

以下需求需在排课模块开发前由老师确认：

1. 一节课的标准时长，是否存在多种时长
2. 是否有小班课，即一节课对应多名学员
3. 学员请假提前多久不扣课时
4. 课时按次购买还是按月，单次通常购买多少节
5. 希望在什么时间收到课程提醒

其中第 2 项影响数据模型：如果存在小班课，`class_sessions` 与学员的关系需从一对一改为多对多，需要增加关联表。这一项优先确认。
