# 开发规范

本文件面向本项目的所有开发工作，包括人工与 AI 辅助编码。产品背景与功能范围见 `plan.md`。

---

## 一、项目定位

英语工作室的排课与学员管理工具，微信小程序。一期只做老师端，单用户使用，不涉及支付与学生端。

技术选型的核心约束是交付周期短（14 天），因此优先选择开箱即用的方案，避免自建后端与过度抽象。

---

## 二、技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 框架 | Taro 3 + React + TypeScript | 保留后续编译 H5 与 App 的能力 |
| UI | NutUI React (Taro 版) | 不引入第二套组件库 |
| 后端 | 微信云开发 · 云函数（Node.js） | 免备案免服务器，云函数即后端 |
| 数据库 | 微信云开发 · 云数据库（文档型） | 仅管理端可读写，一律经云函数 |
| 日期处理 | dayjs | 统一使用，不混用原生 Date 运算 |
| 包管理 | pnpm | |

不引入 Redux 或其他全局状态库。一期数据规模小，React 内置状态加 Supabase 查询足够。

## 三、目录结构

```
src/
  pages/          页面，每页一个目录
  components/     跨页面复用组件
  services/       云函数调用封装层（api.ts 统一封装，各业务一个文件）
  hooks/          自定义 hooks
  utils/          纯函数工具
  types/          TypeScript 类型定义
  constants/      枚举与常量
cloudfunctions/   云函数（后端），每个函数一个目录，_shared 放共享工具
docs/
  plan.md              项目方案
  dev-guide.md         本文件
  scheduling-rules.md  排课规则定义
  schema.md            云数据库集合设计
```

页面组件不直接调用 `wx.cloud.callFunction`，一律通过 `services/` 层（内部走 `services/api.ts`）。
这样后续如果更换后端，改动收敛在一个目录内。

## 四、数据设计原则

### 课时记账

课时余额由 `credit_logs` 表的 `delta` 字段累加得出，不设可直接修改的余额字段。

任何课时变动都必须写入一条流水，包含变动原因与关联的课程 ID。这样出现争议时可以完整回溯每一次扣减。

余额通过云函数 `credits.getBalance(studentId)` 累加 `creditLogs.delta` 得出。如果性能出现问题，再考虑加缓存字段，但缓存字段只能由流水触发更新，不可手工修改。

### 状态枚举

课程状态：`scheduled` / `completed` / `absent` / `cancelled`

流水原因：`purchase` / `attend` / `absent` / `revert`（撤销完成的反向入账）/ `manual`

所有枚举值集中定义在 `src/constants/`，不在业务代码里写字符串字面量。

### 时间处理

数据库统一存 `timestamptz`，前端统一用 dayjs 处理。循环课的规则存 `weekday` 与 `time_of_day`，具体课程实例存完整时间戳。

## 五、排课模块

这是工作量与风险最集中的模块，实现前必须先完成 `docs/scheduling-rules.md`，明确以下问题：

- 循环课一次生成多少周，是一次性生成实例还是按需滚动生成
- 修改单节课时是否影响整个循环系列
- 删除循环规则时已生成的课程如何处理

- 状态流转中哪些操作可逆，各自对课时的影响

## 六、代码约定

### 命名

- 组件文件 PascalCase，其余 camelCase
- 类型定义与表名对应，如 `Student`、`ClassSession`
- 布尔变量以 `is` / `has` / `can` 开头

### 注释

只在两种情况写注释：解释为什么这样做，以及标注临时方案与已知限制。不写复述代码行为的注释。

### 提交

每个功能模块完成后提交一次，message 使用中文：

```
feat: 学员管理页面
fix: 修复循环课跨月生成错误
refactor: 抽取课时计算逻辑
docs: 补充排课规则说明
```

不提交 `.env`、`node_modules`、`dist`。

### 小程序渲染注意

- **`ScrollView` 不能直接设置 `padding`**：webview 渲染模式下不生效，会导致内容溢出、右侧被裁（字段值、卡片描边缺一块）。`padding` 一律放 `ScrollView` **内层的容器**上。
- 手绘描边/图标用背景 SVG data URI 承载（`SketchFrame` / `StatusMark` / `SketchIcon`），weapp 支持；`feTurbulence` 滤镜纹理支持不稳定，不渲染时退化为纯纸色底可接受。

## 七、开发顺序

按以下顺序推进，每个模块完成后确认再继续，避免一次性铺开：

1. 学员管理（列表、新增、编辑、详情）
2. 课时充值（新增课包并写入流水）
3. 排课日历（周视图、单次课、循环课、冲突检测）
4. 上课记录（完成与缺勤，扣减课时）
5. 首页（今日课程、本周统计、课时预警）
6. 订阅消息提醒

## 八、界面文案

一期使用工具类定位，界面文案统一采用「课表」「学员」「课时」「记录」等表述。

不出现「报名」「购买」「课程销售」「培训机构」等指向教育服务与交易的措辞，以确保工具类目审核顺利。二期升级企业主体后不受此限制。

---

## 九、环境配置

`.env` 需包含云开发环境 ID：

```
TARO_APP_CLOUD_ENV=
```

该文件不提交，仓库中保留 `.env.example` 作为模板。云函数所需的 AppSecret 等密钥只存放在
云函数运行环境的配置中，绝不写入小程序代码或仓库。
